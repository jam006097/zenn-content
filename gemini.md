# Zenn記事執筆規約

## 1. 目的と位置づけ

このドキュメントは、Zennに投稿する記事を **一定の品質で安定して執筆・生成すること** を目的とした規約です。
人が執筆する場合だけでなく、**AI（Gemini）による記事生成時の前提条件**としても利用されることを想定しています。

---

## 2. ファイル・ディレクトリ管理

### 2.1 記事ファイル

* 記事は `articles` ディレクトリに配置する
* ファイル名は `[slug].md` とし、URLスラッグとして意味が分かる英語表記を用いる

---

  * 例: `how-to-use-gemini-cli.md`

### 2.2 画像ファイル

* 画像は `images/{slug}/` ディレクトリ配下に配置する

  * 例: `images/how-to-use-gemini-cli/image1.png`
* ファイル名は半角英数字とハイフンのみを使用する

---

## 2.3 サンプルプロジェクト
* サンプルプロジェクトは`sample/`ディレクトリ配下に配置する
* ユーザーが検証時に作成するのでAIが作成する必要はない

## 3. 記事メタ情報（Frontmatter）

各記事の先頭には、以下の Frontmatter を必ず記述する。

```yaml
---
title: "記事のタイトル"
emoji: "✨"
type: "tech" # tech | idea
topics:
  - gemini
  - cli
  - markdown
published: false
---
```

### 各項目の意味

* `title`: 記事内容が一目で分かる具体的なタイトル
* `emoji`: 記事一覧での視認性を高めるための絵文字（1つ）
* `type`:

  * `tech`: 技術解説・実装・設計に関する記事
  * `idea`: 考え方・学習方法・ノウハウ共有
* `topics`: 記事内容に関連するタグ（3〜5個、最大5個）
* `published`: 公開状態の制御（下書きは `false`）

---

## 4. 執筆スタイル・フォーマット

### 4.1 文体・表記

* 文体は「です・ます調」に統一する
* 英数字は半角、カタカナは全角を使用する
* 技術用語・固有名詞は公式表記に従い、表記ゆれを避ける
* 記事内のコードはコメントを付けて読者が読みやすようにする
* 記事内のコードのコメントは「なぜ」がわかる内容とすること

### 4.2 Markdown運用ルール

* 記事タイトルは `#`（h1）を使用する
* 本文の見出しは `##`（h2）から始める
* コードブロックには必ず言語指定を行う

```ts
console.log("Hello, Zenn!")
```

* Zenn 独自記法（メッセージブロック、アコーディオン等）は可読性向上の目的で使用する

---

## 5. 図表（Mermaid 記法）運用ルール

### 5.1 基本方針

* Zenn は ```mermaid 記法に対応している
* 図は **文章理解を補助する目的**でのみ使用する

### 5.2 使用判断の目安

* 処理の流れ・時系列 → `flowchart` / `sequenceDiagram`
* 構造・責務・関係性 → `classDiagram` / `erDiagram`
* 同じ説明を文章で繰り返している場合は図の追加を検討する

### 5.3 記述ルール

* 1図1メッセージを原則とする
* ノード名・ラベルは日本語または平易な英語を使用する
* 情報を詰め込みすぎない

### 5.4 注意点

* Zenn プレビューで必ず表示確認を行う
* Mermaid の記法バージョン差異に注意する
* 図の意図は本文中で必ず説明する

## 6. AI（Gemini）による記事生成ガイドライン

### 6.1 AIに与える前提条件

* 役割: 経験豊富なソフトウェアエンジニア
* 目的: Zenn に投稿する技術記事を作成する

### 6.2 想定読者

* 実務で開発を行っているエンジニア
* 初学者向けではないが、前提知識は明示する
* 「なぜそうするのか」を重視する読者
* CLI操作を基本とします。
* UNIX環境を基本とします。
* 開発環境がないことを前提としてください。記事内で開発環境が整えられるようにしてください。

### 6.3 構成原則

* 結論 → 理由 → 具体例 → まとめ
* 見出しで論点が追える構成にする
* 1セクション1テーマを守る

### 6.4 内容品質の指針

* What だけでなく Why を必ず説明する
* トレードオフや制約条件に触れる
* 個人の経験・判断は前提条件を明示する
* 用語・略語は初出時に補足する

### 6.5 コード・技術説明

* コードは最小構成で意図が分かるものにする
* なぜその設計・書き方なのかを文章で説明する
* 可読性・保守性・テスト観点を重視する

### 6.6 NGパターン

* 結論のない長文
* 前提知識が暗黙の説明
* 動くだけで理由のないコード

---

## 7. 執筆・公開フロー

1. 記事作成

```bash
npx zenn new:article
```

2. 執筆

* 本規約に従って本文を作成する

3. プレビュー

```bash
npx zenn preview
```

4. Gitコミット

* 新規: `feat(article): 新規記事「〇〇」を追加`
* 修正: `fix(article): 記事「〇〇」を修正`
* 更新: `docs(article): 記事「〇〇」を更新`

5. 公開

* `main` ブランチにマージ、または `npx zenn publish` を実行
